# 解释
## /odometry
- 这个话题目前没有在fastlivo2发布，**目前猜测是来自ros_bridge**
- fastlio2中里程计输出话题是`/aft_mapped_to_init`
    - 他发布的 pose 就是 camera_init（即 odom）→ aft_mapped（即 base_link） 的变换 

| FAST-LIVO2坐标系 | 标准ROS命名 | 含义 |
| --- | --- | --- |
| `camera_init`	| `odom`	| 雷达开机时刻的 IMU 位置（世界/地图坐标系，固定不动） |
| `aft_mapped` | `base_link`	| 当前时刻雷达/IMU 的实时位置（随机器人运动变化）|


```cpp
#include <rclcpp/rclcpp.hpp>
#include <nav_msgs/msg/odometry.hpp>
#include <tf2_ros/transform_broadcaster.h>
#include <geometry_msgs/msg/transform_stamped.hpp>

class OdomToTFNode : public rclcpp::Node
{
public:
    OdomToTFNode() : Node("odom_to_tf")
    {
        // 1. 创建 TF 广播器（用来发布TF）
        tf_broadcaster_ = std::make_shared<tf2_ros::TransformBroadcaster>(this);
        
        // 2. 订阅里程计话题
        odom_sub_ = this->create_subscription<nav_msgs::msg::Odometry>(
            "/aft_mapped_to_init", 
            10,
            std::bind(&OdomToTFNode::odomCallback, this, std::placeholders::_1));
    }

private:
    void odomCallback(const nav_msgs::msg::Odometry::SharedPtr msg)
    {
        // 3. 创建 TransformStamped 消息
        geometry_msgs::msg::TransformStamped transform;
        
        // 4. 填充消息头
        transform.header.stamp = msg->header.stamp;
        transform.header.frame_id = "odom";           // 父坐标系（原来是 camera_init）
        transform.child_frame_id = "base_link";       // 子坐标系（原来是 aft_mapped）
        
        // 5. 复制位置（Odometry 的 pose 就是子坐标系在父坐标系中的位姿）
        transform.transform.translation.x = msg->pose.pose.position.x;
        transform.transform.translation.y = msg->pose.pose.position.y;
        transform.transform.translation.z = msg->pose.pose.position.z;
        
        // 6. 复制姿态（四元数）
        transform.transform.rotation = msg->pose.pose.orientation;
        
        // 7. 发布 TF
        tf_broadcaster_->sendTransform(transform);
        
        RCLCPP_INFO(this->get_logger(), 
            "发布 TF: odom -> base_link, 位置: %.2f, %.2f, %.2f",
            transform.transform.translation.x,
            transform.transform.translation.y,
            transform.transform.translation.z);
    }
    
    std::shared_ptr<tf2_ros::TransformBroadcaster> tf_broadcaster_;
    rclcpp::Subscription<nav_msgs::msg::Odometry>::SharedPtr odom_sub_;
};

int main(int argc, char** argv)
{
    rclcpp::init(argc, argv);
    rclcpp::spin(std::make_shared<OdomToTFNode>());
    rclcpp::shutdown();
    return 0;
}
```

## RViz和TF
- RViz的fixed frame与TF树的map是同一个坐标系
- 所有显示的数据都会依据==TF树==被变换到这个固定坐标系下再渲染，如果你选其他的坐标系作为fixed frame，那么相同的道理
- 两个物理重合但frame_id不同的坐标系，RViz不能显示哪个没有在TF中出现的名字的数据
    -  即使物理重合，也必须发布一个静态变换，让RViz知道这两个坐标系是重合的
## 重定位本质
- 我们使用的待配准的点云的坐标系是odom，因此变换直接发布map->odom
- 比如我们在RViz显示出来二者点云并不重合，那是因为map->odom的变换不正确，也就是说，我们map->odom的变换无法让odom系的点云和map系的原始点云重合，因此我们认为现在机器人位置是错误的
